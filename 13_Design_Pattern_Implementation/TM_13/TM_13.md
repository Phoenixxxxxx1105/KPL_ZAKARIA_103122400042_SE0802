## Jawaban

Bayangkan Anda memiliki sebuah daftar (`<ul>`) dengan 1000 item (`<li>`). Setiap item, ketika diklik, harus menjalankan fungsi tertentu, misalnya menampilkan teks item tersebut.

**Pendekatan Naif (Tanpa Delegasi):**
Anda akan menambahkan *event listener* ke setiap `<li>` secara individual. Ini bisa dilakukan dengan loop:

```javascript
const listItems = document.querySelectorAll('li');
listItems.forEach(item => {
  item.addEventListener('click', function() {
    console.log(this.textContent);
  });
});
```

**Masalah dari pendekatan ini:**

1. **Performa:** Membuat 1000 *event listener* terpisah memakan memori dan waktu inisialisasi. Semakin banyak elemen, semakin berat bebannya.
2. **Dinamisme:** Jika Anda menambahkan item baru ke dalam daftar setelah halaman dimuat, item baru tersebut **tidak akan memiliki** *event listener*. Anda harus ingat untuk menambahkan *listener* secara manual setiap kali ada item baru.
3. **Pemeliharaan:** Kode menjadi berantakan jika Anda harus mengelola banyak *listener* untuk elemen serupa.

*Event delegation* hadir sebagai solusi elegan untuk masalah-masalah ini.

### Definisi Inti: Apa Itu *Event Delegation*?

**Event delegation** adalah sebuah teknik (atau *design pattern*) dalam JavaScript di mana Anda tidak menambahkan *event listener* ke elemen target secara langsung (misalnya, ke setiap `<li>`), melainkan Anda menambahkan satu *event listener* ke **elemen induk (parent)** yang menaungi elemen-elemen target tersebut (misalnya, ke `<ul>`).

Kemudian, di dalam fungsi *handler* dari *event listener* induk tersebut, Anda memanfaatkan **proses *event bubbling*** untuk menentukan elemen anak mana yang sebenarnya memicu *event*, lalu menjalankan logika yang sesuai untuk elemen anak tersebut.

### Mekanisme di Balik Layar: *Event Bubbling*

Untuk memahami *event delegation*, Anda harus paham konsep ***event bubbling***.

Ketika sebuah *event* (misalnya, klik) terjadi pada sebuah elemen DOM, *event* tersebut tidak hanya berhenti di elemen itu saja. akan "menggelembung" (bubble) naik ke atas melalui hierarki DOM, dari elemen target kemen induknya, lalu ke induk dari induknya, dan seterusnya hingga mencapai objek `document (atau `window`).

**Contoh Struktur DOM:**

```
<div id="container">
  <ul id="">
    <li class="item">Item 1</li    <li class="item">Item 2</li>
    <li class="item">Item 3</li>
  </ul>
</div```

Jika Anda mengklik "Item 2", urutan *event* yang terjadi adalah:
1.  `click` pada `<li>Item 2</li>` (target).
2.  `click` pada `<ul id="list">` (induk).
3.  `click` pada `<div id="container"> (induk dari induk).
4.  `click` pada `document`.

*Event delegation* memanfaatkan fakta bahwa *event* akan sampai ke elemen induk (`<ul>`). Jadi, kita cukup mendengarkan *event* di `<ul>`, dari sana kita bisa mencari tahu `<li>` mana yang diklik.

### Implementasi *Event Delegation*: Langkah demi Langkah

Mari kita selesaikan masalah daftar 1000 item tadi dengan *event delegation*.

**HTML:**
```html
<ul id="myList">
  <li>Item 1</li>
  <li>Item 2</li>
  <li>Item 3</li>
  <!-- ... 997 item lainnya ... -->
</ul>
<button id="addItem">Tambah Item Baru</button>
```

**JavaScript (Dengan Delegasi):**

```javascript
// 1. Ambil elemen INDUK (bukan anak-anaknya)
const list = document.getElementById('myList');

// 2. Tambahkan SATU event listener ke elemen induk
list.addEventListener('click', function(event) {
  // 3. Di dalam handler, gunakan properti 'event.target'
  //    untuk mendapatkan elemen yang benar-benar diklik.
  const clickedElement = event.target;

  // 4. (Penting) Pastikan elemen yang diklik adalah jenis yang kita inginkan.
  //    Misalnya, kita hanya ingin merespon klik pada <li>, bukan pada <ul> atau elemen lain di dalam <li>.
  if (clickedElement.tagName === 'LI') {
    console.log('Kamu mengklik:', clickedElement.textContent);
    // Lakukan aksi spesifik untuk item daftar
    clickedElement.style.backgroundColor = 'yellow';
  }
});

// 5. Bukti dinamisme: Item baru yang ditambahkan OTOMATIS memiliki perilaku klik.
document.getElementById('addItem').addEventListener('click', function() {
  const newItem = document.createElement('li');
  newItem.textContent = 'Item Baru ' + (list.children.length + 1);
 list.appendChild(newItem);
  // TIDAK PERLUambahkan event listener ke newItem!
});
```

**Penjel Kode:**

- `event.target`: Ini adalah properti paling penting. Ia selalu merujuk pada elemen **asli** yang memicu *event* (dalam kasus ini, `<li>` yang diklik).
- `event.currentTarget`: Properti ini merujuk pada elemen tempat *listener* dipasang (dalam kasus ini, `<ul id="myList">`). Berguna jika Anda perlu mengakses elemen induk dari dalam *handler*.
- **Pengecekan `tagName`**: Ini adalah langkah krusial. Tanpa pengecekan ini, jika Anda mengklik area kosong di dalam `<ul>` (bukan `<li>`), kode akan tetap berjalan dan mungkin menghasilkan error atau perilaku yang tidak diinginkan. Anda bisa menggunakan `matches()` method untuk pengecekan yang lebih fleksibel, misalnya `if (clickedElement.matches('li'))` atau `if (clickedElement.matches('.item-class'))`.

### Keuntungan Utama *Event Delegation*

1. **Performa Lebih Baik:** Hanya ada satu *event listener* yang terdaftar, bukan ribuan. Ini mengurangi penggunaan memori dan waktu inisialisasi, terutama pada halaman dengan banyak elemen dinamis.
2. **Penanganan Elemen Dinamis:** Elemen yang ditambahkan ke DOM setelah halaman dimuat **secara otomatis** akan terpengaruh oleh *event listener* induk. Ini adalah keuntungan terbesarnya. Anda tidak perlu rep-repot mengikat ulang *listener*.
3. **Kode Lebih Bersih dan Mudah Dirawat:** Logika untuk menangani *event* pada sekelompok elemen terkonsentrasi di satu tempat (fungsi *handler* induk), bukan tersebar banyak tempat.
4. **Mengurangi Penggunaan Memori:** Lebih sedikit fungsi *handler* yang dibuat dan disimpan di memori.

### Kapan *Event Delegasi* TIDAK Tepat Digunakan?

Meskipun sangat kuat, *event* bukanlah solusi untuk segalanya. Ada beberapa situasi di mana pendekatan langsung lebih:

1. **Event yang Tidak Bubbling:** Beberapa *event* tidak melakukan *bubbling* ke atas. Contoh paling terkenal adalah:
   - `focus` dan `blur` (mereka tidak *bubble*, tetapi memiliki padanan yang *bubble* `focusin` dan `focusout`).
   - `mouseenter` dan `mouseleave` (mereka tidakbubble*).
   - `load`, `unload `scroll `, `resize` (umumnya tidak *bubble* atau tidak relevan untuk didelegasikan).
2. **Kebutuhan `stopPropagation()` yang Sering:** Jika Anda sering `event.stopPropagation()` untuk menghentikan *event* agar tidak naik ke induk, maka *event delegation* akan menjadi rumit atau tidak berfungsi seperti yang diharapkan.
3. **Kontrol yang Sangat Spesifik per Elemen:** Jika setiap elemen anak memerlukan logika *handler* yang sangat berbeda dan kompleks, menulis semua logika itu dalam satu fungsi *handler* besar bisa membuat kode menjadi sulit dibaca dan dirawat.
