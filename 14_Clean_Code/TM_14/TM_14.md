Dari dua kode di bawah ini, mana yang kamu ingin cari masalahnya dan perbaiki di tengah-tengah malam, katakanlah jam 1 malam? Mengapa?

function processUser(user) {
  if (user) {
    if (user.isActive) {
      if (user.hasPermission) {
        return doSomething(user)
      }
    }
  }
  return null
}

function processUser(user) {
  if (!isValidCandidate(user)) return null;
  return doSomething(user);
}

function isValidCandidate(user) {
  return user && user.isActive && user.hasPermission;
}


## Jawaban

saya pilih kode kedua karena auh lebih ramah buat otak manusia, apalagi kalau kamu lagi ngebut *debugging* buat ngejar *deadline* Tubes atau nyiapin bahan buat di-*review* asprak besok paginya.

Ini beberapa alasan teknis kenapa snippet kedua adalah pemenangnya:


### 1. Menghindari "Pyramid of Doom" (Nesting yang Dalam)

Kode pertama pakai pola yang sering disebut *Arrow Anti-Pattern* karena bentuk indentasinya makin lama makin menjorok ke kanan kayak ujung panah. Kalau kodenya cuma tiga baris mungkin masih aman, tapi bayangin kalau di dalam `if` itu ada logika kompleks lain. Kognitif kita dipaksa buat mengingat *state* dari `if` pertama sampai ke-tiga secara bersamaan. Itu bikin capek.


### 2. Penerapan *Guard Clause* (Early Return)

Di Kode kedua:

if (!isValidCandidate(user)) return null;

Ini namanya *Guard Clause* atau  *Early Return* . Konsepnya kayak satpam di depan pintu: kalau syaratnya nggak terpenuhi, langsung tendang keluar (`return null`) di awal. Kodenya jadi *flat* (datar) dan alur eksekusi utamanya (`doSomething`) nggak tertutup oleh tumpukan blok `if`.


### 3. Kode Membaca Seperti Cerita (Descriptive Logic)

Kode kedua mengekstrak kondisinya ke fungsi `isValidCandidate(user)`.

* Di Kode pertama, kita harus mengevaluasi sendiri: "Oh, kalau dia ada, dan aktif, dan punya izin, baru jalan."
* Di Kode kedua, kodenya langsung ngomong ke kita: "Kalau dia bukan kandidat yang valid, batalin. Kalau valid, lakukan sesuatu."

Logika bisnisnya terbungkus rapi, jadi kalau suatu saat syarat "valid"-nya nambah (misal: harus  *premium user* ), kamu cukup ubah fungsi `isValidCandidate` tanpa perlu menyentuh fungsi `processUser`. Kodenya jauh lebih  *maintainable* .
